# 特性规范 (Feature Specification): 数据治理知识库重构

**特性名称**: 数据治理知识库规范
**创建日期**: 2026-02-06
**状态**: 草稿 (Draft)
**输入**: 用户指令 - 重构 /数据治理 目录，遵循新章程。

## 用户场景与测试 (User Scenarios & Testing)

### 用户故事 1 - 读者浏览知识体系 (Priority: P1)

作为一名数据治理学习者，我希望能够通过清晰、扁平的目录结构快速找到所需知识，而不需要在深层嵌套的文件夹中迷失。

**优先级理由**: 知识库的核心价值在于传播，易用性是第一位的。扁平化结构能降低认知负荷。

**独立测试**: 
- 打开根目录，仅看到一级章节目录。
- 进入任意一级目录，能直接看到所有相关的 Markdown 文件，无需进入二级子目录。

**验收场景**:
1. **Given** 用户在 `/数据治理` 根目录, **When** 查看文件列表, **Then** 只能看到如 `01.xxx`, `02.xxx` 等一级目录和 `index.md`。
2. **Given** 用户进入 `01.数据治理核心概念` 目录, **When** 寻找 "数据治理定义", **Then** 直接看到 `1.1-数据治理定义.md` 文件，而不是 `第一章/定义.md` 文件夹。

---

### 用户故事 2 - 贡献者提交新内容 (Priority: P1)

作为一名内容贡献者，我希望有明确的文件命名和存放规则，以便我知道将新写的文章放在哪里，且确信不会破坏现有的结构。

**独立测试**: 
- 创建一个新文件，按照规范命名（英文/拼音），放入对应的一级目录。
- 检查 URL 引用是否简洁且无编码问题。

**验收场景**:
1. **Given** 贡献者写了一篇关于 "Data Quality" 的文章, **When** 保存文件, **Then** 文件名被命名为 `data_quality.md` (或 `shuju_zhiliang.md`)，而不是 `数据质量.md`。
2. **Given** 贡献者需要插入一张流程图, **When** 引用图片, **Then** 图片路径为当前目录下的 `images/flowchart.png`，而不是引用外部链接。

## 功能需求 (Requirements)

### 目录结构规范 (Directory Structure)

- **FR-001**:  知识库**必须**采用扁平化结构，最大目录深度为 3 层（根目录 -> 一级章节目录 -> 文件）。
- **FR-002**:  仅**第 1 层**（根目录下）允许创建章节文件夹（格式：`序号.章节名`，如 `01.Overview`）。
- **FR-003**:  **严禁**在第 1 层目录下创建业务内容的子目录（第 2/3 层内容必须是 `.md` 文件）。
- **FR-004**:  每个一级目录下**必须**包含一个 `index.md` 文件，作为该章节的导航页。
- **FR-005**:  资源文件（图片、附件）**必须**存放在同级的 `images/` 或 `resources/` 目录中。

### 文件命名规范 (Naming Conventions)

- **FR-006**:  所有 Markdown 文件名（不含目录名）**必须**仅包含：英文字母、数字、下划线 `_` 或连字符 `-`。
- **FR-007**:  **严禁**在 Markdown 文件名中使用中文字符（`index.md` 和 `README.md` 除外）。
- **FR-008**:  文件名应具有描述性，建议使用英文术语（如 `data_lineage.md`）。

### 内容标准 (Content Standards)

- **FR-009**:  所有核心概念阐述**必须**包含 at least 一张图表（Mermaid 或 图片引用）。
- **FR-010**:  文档内链接**必须**使用相对路径（如 `./images/pic.png`），禁止使用绝对路径或 `file://` 协议。

### 内容格式规范 (Formatting Standards)

- **FR-012**:  **语义强化**：所有关键技术术语在章节中首次出现时必须使用 Bold (`**Term**`)。
- **FR-013**:  **提示标识**：所有 Note/Warning 类信息必须使用 Italic (`_Note content_`) 或 GitHub Alert 语法。
- **FR-014**:  **列表排版**：嵌套列表**必须**使用 4 空格缩进（Google Style）。
- **FR-015**:  **标题字体**：虽然 Markdown 不直接控制字体，但如果生成 HTML 预览，必须确保标题渲染为宋体 (SimSun)，正文为微软雅黑 (Microsoft YaHei)。

### 工程规范 (Engineering Standards)

- **FR-011**:  所有用于维护、生成或重构知识库的代码脚本（Python/Shell）**必须**存放在 `/datapipeline/` 目录下。

## 成功标准 (Success Criteria)

### 可衡量结果 (Measurable Outcomes)

- **SC-001**:  利用脚本扫描 `/数据治理` 目录，层级深度 > 3 的目录数量为 0。
- **SC-002**:  利用脚本扫描所有 `.md` 文件名，包含非 ASCII 字符（中文）的文件数量为 0（排除 `index.md` 等白名单）。
- **SC-003**:  所有 `index.md` 文件均包含指向同级所有 `.md` 文件的有效链接。
- **SC-004**:  随机抽取 5 篇核心概念文档，配图率为 100%。
